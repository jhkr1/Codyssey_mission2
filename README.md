# 나만의 퀴즈 게임

## 1. 프로젝트 개요

이 프로젝트는 Python으로 만든 콘솔 기반 퀴즈 게임이다.  
사용자는 메뉴를 통해 퀴즈를 풀고, 새로운 퀴즈를 추가하고, 등록된 퀴즈 목록을 확인하고, 최고 점수를 확인하고, 퀴즈를 삭제할 수 있다.  
프로그램을 종료해도 퀴즈와 최고 점수가 유지되도록 프로젝트 루트의 `state.json` 파일에 데이터를 저장한다.

이 README는 단순한 소개 문서가 아니라, 이 과제를 통해 배운 내용을 정리한 기록서다.  
따라서 "무엇을 만들었는가"만 적지 않고, "왜 이렇게 구현했는가", "이 구현으로 어떤 결과가 나오는가", "이 과정에서 어떤 Python 개념을 이해하게 되었는가"까지 함께 기록한다.

## 2. 퀴즈 주제 선정 이유

퀴즈 주제는 `Python 기초 문법`이다.

이 주제를 선택한 이유는 다음과 같다.

- 프로젝트를 구현하면서 동시에 Python 핵심 개념을 복습할 수 있다.
- 변수, 조건문, 반복문, 함수, 클래스 같은 기본 개념을 문제로 다시 정리할 수 있다.
- 학습이 진행될수록 새로운 퀴즈를 직접 추가하며 프로젝트를 확장하기 쉽다.
- "프로그램을 만들면서 언어를 배운다"는 미션 의도와 가장 잘 맞는다.

## 6. 코드 구조를 이렇게 잡은 이유

이번 버전은 코드 리뷰를 쉽게 하기 위해 구조를 일부러 단순하게 유지했다.  
클래스는 `Quiz`, `QuizGame` 두 개만 두고, 나머지 기능은 `QuizGame`의 메서드로 정리했다.

- `Quiz`: 문제 1개를 표현하는 클래스
- `QuizGame`: 메뉴, 입력 처리, 게임 진행, 저장/불러오기, 점수 갱신을 관리하는 클래스

이렇게 구성한 이유는 다음과 같다.

- 초보자가 읽을 때 파일을 너무 많이 넘나들지 않아도 된다.
- "문제 하나"와 "게임 전체"라는 책임 구분이 명확하다.
- 코드 리뷰 때 `main.py -> QuizGame.run() -> 각 기능 메서드` 순서로 설명하기 쉽다.
- 기능은 분리하되, 과도한 모듈 분리는 줄여서 전체 흐름을 놓치지 않게 했다.

즉 이 프로젝트의 목표는 "복잡하게 잘게 나누는 것"이 아니라, "흐름이 보이게 구조화하는 것"이다.

## 7. 프로그램 전체 흐름

프로그램은 아래 순서로 동작한다.

1. `main.py`가 실행된다.
2. `QuizGame()` 객체가 생성된다.
3. `QuizGame.__init__()` 안에서 `load_state()`가 호출된다.
4. `state.json`이 있으면 저장된 퀴즈와 점수를 불러온다.
5. 파일이 없거나 손상되었으면 기본 퀴즈 데이터로 복구한다.
6. `run()` 메서드가 메뉴를 반복해서 출력한다.
7. 사용자는 메뉴 번호를 입력한다.
8. 입력한 번호에 따라 퀴즈 풀기, 추가, 목록, 점수 확인, 삭제, 종료 기능이 실행된다.
9. 데이터가 바뀌면 `save_state()`가 현재 상태를 다시 `state.json`에 저장한다.

이 흐름을 기준으로 코드를 읽으면 프로그램 전체 동작이 훨씬 쉽게 보인다.

## 8. 핵심 클래스 설명

### 8-1. `Quiz` 클래스

`Quiz` 클래스는 퀴즈 한 문제를 표현한다.

속성:

- `question`: 문제 문장
- `choices`: 선택지 4개를 담은 리스트
- `answer`: 정답 번호

메서드:

- `display()`: 문제와 선택지를 출력한다.
- `is_correct()`: 사용자의 입력이 정답인지 확인한다.
- `correct_choice_text()`: 정답 번호에 해당하는 실제 선택지 문장을 가져온다.
- `to_dict()`: `Quiz` 객체를 JSON 저장용 딕셔너리로 바꾼다.
- `from_dict()`: 딕셔너리 데이터를 다시 `Quiz` 객체로 복원한다.

이 클래스가 필요한 이유는 "문제 데이터"와 "문제 관련 동작"을 묶기 위해서다.  
문제를 단순히 문자열 3개로 흩어 놓지 않고, 하나의 객체로 다루면 코드 의미가 훨씬 분명해진다.

### 8-2. `QuizGame` 클래스

`QuizGame` 클래스는 게임 전체를 관리한다.

속성:

- `quizzes`: 현재 등록된 퀴즈 목록
- `best_score`: 최고 점수
- `best_correct_count`: 최고 점수 당시 맞힌 개수
- `best_total_count`: 최고 점수 당시 전체 문제 수
- `state_path`: 저장 파일 경로

주요 메서드:

- `run()`: 메인 메뉴 반복 실행
- `show_menu()`: 메뉴 출력
- `play_quiz()`: 퀴즈 진행
- `add_quiz()`: 퀴즈 추가
- `show_quiz_list()`: 퀴즈 목록 출력
- `show_best_score()`: 최고 점수 출력
- `delete_quiz()`: 퀴즈 삭제
- `prompt_text()`, `prompt_number()`: 입력 검증
- `load_state()`, `save_state()`: 파일 저장/불러오기

즉 `QuizGame`은 "게임 운영자" 같은 클래스다.  
반면 `Quiz`는 "문제 1개" 자체를 표현한다.

### 8-3. 함수와 클래스를 언제 사용하는가

이 프로젝트를 만들면서 함수와 클래스를 아무 때나 섞어 쓴 것은 아니다.  
"하나의 동작만 깔끔하게 실행하면 되는가", 아니면 "데이터와 동작을 함께 묶어야 하는가"를 기준으로 나눴다.

함수를 사용하기 좋은 경우:

- 하나의 작업을 반복해서 재사용할 때
- 입력값을 받아 결과를 만들거나, 특정 동작 하나를 수행할 때
- 상태를 길게 저장할 필요 없이 짧고 명확한 기능이 필요할 때

예를 들어 `prompt_text()`, `prompt_number()` 같은 메서드는 결국 "입력을 검사한다"는 기능이 핵심이다.  
이런 로직은 같은 규칙을 여러 곳에서 재사용하므로 따로 빼는 것이 좋다.

클래스를 사용하기 좋은 경우:

- 관련된 데이터와 동작을 함께 묶어야 할 때
- 프로그램 안에서 하나의 대상을 계속 관리해야 할 때
- 여러 함수가 같은 상태를 공유해야 할 때

이 프로젝트에서 `Quiz`는 문제 문장, 선택지, 정답이라는 데이터를 가지고 있고, 그 데이터와 관련된 `display()`, `is_correct()`, `to_dict()` 같은 동작도 함께 가진다.  
또 `QuizGame`은 퀴즈 목록, 최고 점수, 저장 경로 같은 상태를 계속 유지하면서 게임 흐름 전체를 관리한다.

즉 이렇게 이해하면 된다.

- 함수: "한 가지 일을 처리하는 도구"
- 클래스: "데이터와 그 데이터를 다루는 기능을 함께 묶은 설계도"

정리하면, 단순히 한 번 계산하거나 입력을 검사하는 정도라면 함수가 더 알맞고, 문제 객체나 게임 전체처럼 상태와 책임이 함께 움직이면 클래스를 사용하는 편이 더 자연스럽다.

## 9. 왜 정답을 숫자(1~4)로 저장했는가

이 프로젝트에서는 정답을 문자열이 아니라 숫자 `1~4`로 저장한다.

예:

```python
Quiz(
    "파이썬의 창시자는 누구인가요?",
    ["귀도 반 로섬", "리누스 토르발스", "제임스 고슬링", "브렌던 아이크"],
    1,
)
```

이렇게 한 이유는 다음과 같다.

- 사용자는 메뉴에서 숫자로 답을 입력한다.
- 선택지도 1번, 2번, 3번, 4번처럼 번호로 출력된다.
- 따라서 비교 기준도 숫자로 맞추는 것이 가장 단순하다.
- 문자열 정답을 직접 비교하면 오타, 공백, 대소문자 같은 문제를 더 많이 신경 써야 한다.

결과적으로 정답 판별 로직이 단순해진다.

```python
def is_correct(self, user_answer):
    return user_answer == self.answer
```

이 메서드는 "사용자 답 == 정답 번호"만 확인하면 되므로 읽기 쉽고 오류 가능성도 줄어든다.

## 10. `to_dict()` 와 `from_dict()`를 왜 만들었는가

이 부분은 이 프로젝트에서 가장 중요한 학습 포인트 중 하나다.

### 10-1. 문제 상황

`Quiz`는 사용자 정의 클래스다.  
하지만 JSON 파일은 기본적으로 문자열, 숫자, 리스트, 딕셔너리 같은 기본 자료형만 저장할 수 있다.

즉, 아래와 같은 객체를 바로 JSON에 저장할 수는 없다.

```python
quiz = Quiz("문제", ["A", "B", "C", "D"], 2)
```

JSON은 "객체가 어떤 클래스인지"를 모른다.  
그래서 저장할 때는 `Quiz` 객체를 JSON이 이해할 수 있는 `dict` 형태로 바꿔야 한다.

### 10-2. `to_dict()`의 역할

```python
def to_dict(self):
    return {
        "question": self.question,
        "choices": self.choices,
        "answer": self.answer,
    }
```

이 메서드는 `Quiz` 객체를 딕셔너리로 바꾼다.

왜 필요한가:

- `json.dump()`는 사용자 정의 객체를 그대로 저장하지 못한다.
- 그래서 JSON이 이해할 수 있는 자료형으로 바꾸는 과정이 필요하다.
- `Quiz` 안에서 직접 변환하게 하면 변환 기준이 한곳에 모인다.

결과:

- 저장할 때 모든 퀴즈를 `[quiz.to_dict() for quiz in self.quizzes]`처럼 쉽게 처리할 수 있다.
- 저장 형식이 일관되게 유지된다.
- 나중에 구조가 바뀌더라도 `to_dict()`만 수정하면 대응하기 쉽다.

### 10-3. `from_dict()`의 역할

파일에서 JSON을 읽으면 다시 `Quiz` 객체가 나오는 것이 아니라, 딕셔너리 데이터가 나온다.

예를 들어 `json.load()` 결과는 이런 형태다.

```python
{
    "question": "파이썬의 창시자는 누구인가요?",
    "choices": ["귀도 반 로섬", "리누스 토르발스", "제임스 고슬링", "브렌던 아이크"],
    "answer": 1
}
```

하지만 게임 안에서는 딕셔너리보다 `Quiz` 객체가 더 다루기 편하다.  
왜냐하면 `display()`, `is_correct()`, `correct_choice_text()` 같은 메서드를 바로 쓸 수 있기 때문이다.

그래서 딕셔너리를 다시 객체로 바꾸는 메서드가 필요하다.

```python
@classmethod
def from_dict(cls, data):
    ...
    return cls(question.strip(), cleaned_choices, answer)
```

왜 필요한가:

- 저장된 JSON 데이터를 다시 `Quiz` 객체로 복원하기 위해
- 게임 로직에서 객체 메서드를 계속 사용할 수 있게 하기 위해
- 딕셔너리 검증과 객체 생성 규칙을 한곳에 모으기 위해

결과:

- `load_state()`에서 저장 데이터를 읽은 뒤 바로 `Quiz` 객체 목록으로 바꿀 수 있다.
- 이후 게임 코드는 JSON 내부 구조를 몰라도 된다.
- 즉, 저장 형식과 게임 로직이 적당히 분리된다.

### 10-4. `self.quizzes = [Quiz.from_dict(item) for item in quizzes_data]`는 무슨 뜻인가

저장된 데이터를 불러올 때 아래 코드가 나온다.

```python
self.quizzes = [Quiz.from_dict(item) for item in quizzes_data]
```

이 코드는 한 줄이지만 실제로는 여러 단계를 한 번에 처리한다.

먼저 `quizzes_data`는 `json.load()`로 읽어 온 퀴즈 딕셔너리 목록이다.

예를 들면 이런 형태다.

```python
[
    {
        "question": "파이썬의 창시자는 누구인가요?",
        "choices": ["귀도 반 로섬", "리누스 토르발스", "제임스 고슬링", "브렌던 아이크"],
        "answer": 1
    },
    {
        "question": "파이썬에서 리스트를 만들 때 사용하는 기호는 무엇인가요?",
        "choices": ["()", "{}", "[]", "<>"],
        "answer": 3
    }
]
```

하지만 게임 안에서 필요한 것은 딕셔너리 목록이 아니라 `Quiz` 객체 목록이다.  
그래서 리스트 안의 각 딕셔너리를 하나씩 꺼내 `Quiz.from_dict(item)`으로 바꾼다.

즉 이 한 줄은 아래 반복문과 같은 뜻이다.

```python
converted_quizzes = []

for item in quizzes_data:
    quiz = Quiz.from_dict(item)
    converted_quizzes.append(quiz)

self.quizzes = converted_quizzes
```

여기서 각 부분의 의미는 다음과 같다.

- `quizzes_data`: 저장 파일에서 읽은 딕셔너리 목록
- `for item in quizzes_data`: 딕셔너리를 하나씩 꺼냄
- `Quiz.from_dict(item)`: 그 딕셔너리를 `Quiz` 객체로 변환
- `[ ... ]`: 변환된 결과를 새 리스트로 만듦
- `self.quizzes =`: 완성된 객체 리스트를 게임 상태에 저장

### 10-5. 왜 이렇게 작성했는가

이 표현은 Python의 `list comprehension`(리스트 컴프리헨션) 문법이다.

왜 사용했는가:

- "각 딕셔너리를 `Quiz` 객체로 바꿔서 새 리스트를 만든다"는 의도가 한 줄에 분명하게 보인다.
- 반복문보다 짧지만 의미는 그대로 유지된다.
- `load_state()`에서 가장 중요한 작업이 "복원"이라는 점이 눈에 잘 들어온다.

즉 이 코드는 "저장된 퀴즈 데이터 전체를 프로그램이 사용할 수 있는 객체 목록으로 복원한다"는 뜻이다.

### 10-6. 이 코드가 중요한 이유

이 줄이 없으면 `self.quizzes` 안에는 딕셔너리만 들어가게 된다.  
그러면 아래 같은 메서드를 사용할 수 없다.

- `quiz.display()`
- `quiz.is_correct()`
- `quiz.correct_choice_text()`

딕셔너리는 키로 값을 꺼낼 수는 있지만, `Quiz` 클래스의 메서드는 가지고 있지 않기 때문이다.

그래서 이 한 줄은 단순한 변환이 아니라, 저장 파일 속 데이터를 다시 "게임 안의 객체 세계"로 되돌리는 핵심 단계다.

## 11. `@classmethod`를 왜 사용했는가

`from_dict()` 앞에는 `@classmethod`가 붙어 있다.

```python
@classmethod
def from_dict(cls, data):
```

이 의미는 "이 메서드는 특정 객체가 아니라 클래스 기준으로 호출된다"는 뜻이다.

일반 메서드는 `self`를 받는다.

```python
def to_dict(self):
```

이건 이미 존재하는 객체를 기준으로 동작한다.

반면 `from_dict()`는 아직 객체가 없는 상태에서 새 객체를 만들어야 한다.  
그러므로 `self`가 아니라 `cls`를 사용한다.

### 11-1. 왜 `self`가 아니라 `cls`를 쓰는가

`self`는 "이미 만들어진 객체 자신"을 가리킨다.  
즉 `self`가 등장하는 메서드는 기본적으로 객체가 먼저 존재해야 호출할 수 있다.

예를 들어 아래 메서드는 이미 만들어진 `Quiz` 객체가 있어야만 사용할 수 있다.

```python
quiz = Quiz("문제", ["A", "B", "C", "D"], 2)
quiz.to_dict()
```

하지만 `from_dict()`는 반대 상황이다.  
이 메서드는 아직 `Quiz` 객체가 없는 상태에서 시작한다.

우리는 파일에서 읽은 딕셔너리 데이터를 가지고 "새 `Quiz` 객체를 만들어야" 한다.

즉 흐름이 이렇다.

1. 딕셔너리 데이터가 있다.
2. 그 딕셔너리로 새 객체를 만들어야 한다.
3. 그러므로 기존 객체(`self`)가 아니라 클래스 자체(`cls`)를 기준으로 작업해야 한다.

그래서 `from_dict()`는 인스턴스 메서드가 아니라 클래스 메서드가 된다.

### 11-2. `@classmethod`가 실제로 하는 일

`@classmethod`를 붙이면 Python은 이 메서드의 첫 번째 인자로 객체가 아니라 클래스를 넘겨준다.

```python
@classmethod
def from_dict(cls, data):
    return cls(...)
```

여기서 `cls`는 `Quiz` 클래스를 가리킨다.

즉 아래처럼 호출하면:

```python
quiz = Quiz.from_dict(data)
```

Python은 내부적으로 대략 이런 느낌으로 이해하면 된다.

```python
quiz = Quiz.from_dict(data)
# from_dict 안에서 cls == Quiz
```

그래서 `return cls(...)`는 결국 `return Quiz(...)`와 같은 역할을 한다.

### 11-3. 왜 `from_dict()`에 특히 잘 맞는가

`from_dict()`의 목적은 "기존 객체를 수정하는 것"이 아니라 "새 객체를 생성하는 것"이다.  
이런 메서드는 클래스 메서드와 아주 잘 맞는다.

왜냐하면:

- 아직 객체가 없는데도 호출할 수 있다.
- 클래스 이름으로 직접 호출하는 것이 자연스럽다.
- "딕셔너리 -> 객체 생성"이라는 역할이 메서드 이름만 봐도 잘 드러난다.

즉 `Quiz.from_dict(data)`는 읽는 사람 입장에서 의미가 아주 분명하다.

- `Quiz`: 어떤 객체를 만들지 알려 줌
- `from_dict`: 무엇으로부터 만들지 알려 줌

코드가 거의 문장처럼 읽힌다.

### 11-4. 그냥 일반 함수로 만들면 안 되는가

물론 가능하다. 예를 들면 이렇게 쓸 수도 있다.

```python
def make_quiz_from_dict(data):
    return Quiz(
        data["question"],
        data["choices"],
        data["answer"],
    )
```

하지만 이 방식은 몇 가지 아쉬운 점이 있다.

- 객체 생성 규칙이 클래스 바깥으로 흩어진다.
- `Quiz`를 만드는 로직이 `Quiz` 클래스 안에 있지 않아서 응집도가 떨어진다.
- 나중에 생성 규칙이 바뀌면 관련 함수를 따로 찾아 수정해야 할 수 있다.

반면 `Quiz.from_dict()`로 두면 "Quiz를 만드는 방법"이 `Quiz` 클래스 안에 들어간다.  
즉 데이터와 그 데이터를 생성하는 규칙이 한곳에 모인다.

이게 객체지향에서 중요한 장점이다.

### 11-5. `@classmethod`를 쓰면 좋은 이유

왜 `cls`를 쓰는가:

- `Quiz.from_dict(data)`처럼 클래스에서 직접 호출할 수 있다.
- 내부에서 `cls(...)`를 호출해 새 객체를 생성할 수 있다.
- 나중에 클래스를 상속해서 확장하더라도 유연하게 재사용할 수 있다.

즉 `from_dict()`는 "저장된 딕셔너리로부터 새 Quiz 객체를 만들어 주는 보조 생성 메서드"라고 이해하면 된다.

조금 더 풀어 쓰면 장점은 다음과 같다.

- 생성 방식이 클래스 안에 모인다.  
  객체 생성 규칙을 클래스 바깥 함수에 흩어 놓지 않아도 된다.
- 이름이 명확하다.  
  `Quiz.from_dict(data)`는 "딕셔너리로부터 Quiz를 만든다"는 의미가 바로 보인다.
- 유지보수가 쉽다.  
  나중에 필드가 늘어나거나 검증 규칙이 바뀌어도 `from_dict()`만 수정하면 된다.
- 저장/복원 패턴에 잘 맞는다.  
  `to_dict()`와 짝을 이루어 읽기 좋은 구조가 된다.
- 상속에 유리하다.  
  `cls(...)`를 사용하면 나중에 `Quiz`를 상속한 클래스에서도 같은 메서드를 더 자연스럽게 재사용할 수 있다.

### 11-6. `cls(...)`가 중요한 이유

많은 초보자가 여기서 "그냥 `Quiz(...)`라고 쓰면 되지 않나?"라고 생각한다.  
현재 프로젝트만 보면 실제로 그렇게 써도 동작할 수 있다.

하지만 `cls(...)`를 쓰면 더 일반적인 코드가 된다.

예를 들어 나중에 `AdvancedQuiz`라는 클래스를 만든다고 가정해 보자.

```python
class AdvancedQuiz(Quiz):
    pass
```

이때 `from_dict()`가 `cls(...)`를 사용하면:

```python
AdvancedQuiz.from_dict(data)
```

호출 시 `cls`는 `AdvancedQuiz`가 된다.  
즉 같은 메서드를 재사용해도 결과로 `AdvancedQuiz` 객체가 만들어질 수 있다.

반대로 메서드 안에서 `Quiz(...)`를 직접 적어 버리면 항상 `Quiz`만 생성된다.  
이 차이가 클래스 메서드의 유연함이다.

지금 과제에서는 상속을 직접 사용하지 않더라도, Python에서 왜 `cls`를 쓰는지 이해하는 좋은 연습이 된다.

### 11-7. 이 프로젝트에서 `@classmethod`를 사용한 최종 이유

정리하면 `from_dict()`에 `@classmethod`를 사용한 이유는 다음과 같다.

- `from_dict()`는 기존 객체를 다루는 메서드가 아니라 새 객체를 만드는 메서드이기 때문이다.
- 딕셔너리 데이터를 `Quiz` 객체로 복원하는 책임을 클래스 안에 두고 싶었기 때문이다.
- `Quiz.from_dict(data)`라는 형태가 의미상 가장 자연스럽고 읽기 쉽기 때문이다.
- 나중에 구조 변경이나 확장에도 더 유연하게 대응할 수 있기 때문이다.

즉 `@classmethod`는 단순한 문법 장식이 아니라, "객체를 만드는 책임"을 클래스에 두기 위한 설계 선택이다.

## 12. `from_dict()` 안의 검증을 왜 이렇게 했는가

`from_dict()`는 단순히 딕셔너리 값을 꺼내는 역할만 하지 않는다.  
읽어 온 데이터가 정말 올바른 퀴즈 데이터인지 검사하는 역할도 함께 한다.

이때 자주 등장하는 함수가 `isinstance()`다.

```python
isinstance(data, dict)
isinstance(question, str)
isinstance(answer, int)
```

### 12-0. `isinstance()`는 무엇인가

`isinstance(값, 자료형)`은 "이 값이 이 자료형에 해당하는가?"를 확인하는 함수다.

예:

```python
isinstance("hello", str)   # True
isinstance(3, int)         # True
isinstance([1, 2], list)   # True
isinstance({"a": 1}, dict) # True
```

반대로 자료형이 다르면 `False`가 나온다.

```python
isinstance("3", int)       # False
isinstance(3, str)         # False
```

왜 이 함수가 중요한가:

- JSON 파일에서 읽은 데이터가 내가 기대한 형식인지 확인할 수 있다.
- 문자열이어야 하는 자리에 숫자가 들어왔는지 검사할 수 있다.
- 리스트여야 하는 자리에 딕셔너리가 들어왔는지 검사할 수 있다.

이 프로젝트에서는 저장된 데이터를 무조건 믿지 않고, `isinstance()`로 자료형부터 확인한 뒤에만 사용한다.

즉 `isinstance()`는 "데이터의 형태를 확인하는 안전장치"라고 이해하면 된다.

### 12-0-1. 왜 `type()` 대신 `isinstance()`를 썼는가

초보자 입장에서는 `type(question) == str`처럼 써도 되지 않나 생각할 수 있다.  
물론 간단한 경우에는 동작할 수 있다. 하지만 Python에서는 보통 `isinstance()`를 더 많이 사용한다.

이유는 다음과 같다.

- 의미가 더 직접적이다.  
  "이 값이 문자열인가?"를 그대로 읽을 수 있다.
- 여러 자료형 검사가 더 자연스럽다.
- 상속 구조가 있는 클래스에서도 더 유연하게 동작한다.

지금 프로젝트에서는 상속을 직접 쓰지는 않지만, Python다운 방식으로 자료형을 검사하는 습관을 익히기 위해 `isinstance()`를 사용했다.

```python
if not isinstance(data, dict):
    raise ValueError("퀴즈 항목은 딕셔너리여야 합니다.")
```

이 검사는 "퀴즈 한 개의 데이터는 반드시 딕셔너리여야 한다"는 규칙을 확인한다.

예를 들어 저장 파일이 아래처럼 잘못되어 있으면 문제가 생긴다.

```json
"quizzes": ["문자열", 123, true]
```

이런 데이터는 퀴즈로 사용할 수 없기 때문에 초기에 걸러야 한다.

### 12-1. 문제(question) 검증

```python
if not isinstance(question, str) or not question.strip():
    raise ValueError("문제 데이터가 올바르지 않습니다.")
```

이 검사의 의미:

- 문제는 문자열이어야 한다.
- 공백만 있는 문자열은 허용하지 않는다.

왜 필요한가:

- 숫자, 리스트, 빈 문자열 같은 값은 문제 문장으로 사용할 수 없다.
- `strip()`을 사용하면 `"   "` 같은 입력도 빈 값으로 판단할 수 있다.

결과:

- 화면에 이상한 문제 문장이 출력되는 상황을 막는다.

### 12-2. 선택지(choices) 검증

```python
if not isinstance(choices, list) or len(choices) != 4:
    raise ValueError("선택지 데이터가 올바르지 않습니다.")
```

이 검사의 의미:

- 선택지는 리스트여야 한다.
- 선택지는 정확히 4개여야 한다.

왜 필요한가:

- 이 프로젝트의 UI와 정답 번호 체계는 4지선다를 기준으로 만들어져 있다.
- 선택지 개수가 달라지면 출력 로직과 정답 입력 범위가 깨질 수 있다.

결과:

- 프로그램 전체 규칙이 일관되게 유지된다.

다음 검사는 선택지 내부 값까지 확인한다.

```python
if not all(isinstance(choice, str) and choice.strip() for choice in choices):
    raise ValueError("선택지 데이터가 올바르지 않습니다.")
```

이 검사의 의미:

- 선택지 4개가 모두 문자열이어야 한다.
- 각 선택지가 비어 있으면 안 된다.

`all(...)`은 "모든 값이 조건을 만족할 때만 True"가 된다.  
즉 선택지 중 하나라도 빈 문자열이거나 문자열이 아니면 오류로 처리된다.

결과:

- 화면에 비어 있는 선택지가 출력되는 일을 막는다.
- 데이터 품질이 일정하게 유지된다.

### 12-3. 정답(answer) 검증

```python
if not isinstance(answer, int) or not 1 <= answer <= 4:
    raise ValueError("정답 데이터가 올바르지 않습니다.")
```

이 검사의 의미:

- 정답은 정수여야 한다.
- 정답 번호는 1에서 4 사이여야 한다.

왜 필요한가:

- 정답이 문자열 `"1"`이면 숫자 비교가 어긋날 수 있다.
- 정답이 7이면 존재하지 않는 선택지를 가리키게 된다.

결과:

- 정답 판별과 정답 문구 출력이 안전해진다.

## 13. `strip()`을 왜 사용했는가

`from_dict()` 마지막 부분에는 아래 코드가 있다.

```python
cleaned_choices = [choice.strip() for choice in choices]
return cls(question.strip(), cleaned_choices, answer)
```

이 코드는 문자열 앞뒤 공백을 제거한다.

왜 필요한가:

- 사용자가 `" 파이썬 "`처럼 앞뒤에 공백을 넣고 저장했을 수 있다.
- 저장 파일을 직접 수정하다가 불필요한 공백이 들어갈 수도 있다.
- 공백 때문에 데이터는 "있는 것처럼 보이지만", 실제 출력 품질은 나빠질 수 있다.

결과:

- 문제와 선택지 문장이 깔끔하게 저장된다.
- `"파이썬"`과 `" 파이썬 "`을 사실상 같은 값처럼 다룰 수 있다.

즉 `strip()`은 단순한 꾸밈이 아니라, 데이터 품질을 정리하는 정제 단계다.

## 14. 저장과 불러오기에서 이 구조가 만드는 장점

이 프로젝트의 저장/복원 흐름은 아래처럼 정리할 수 있다.

### 저장할 때

1. 게임 안에는 `Quiz` 객체 목록이 들어 있다.
2. `save_state()`가 각 객체에 `to_dict()`를 호출한다.
3. 딕셔너리 목록을 JSON으로 저장한다.

### 불러올 때

1. `load_state()`가 `json.load()`로 딕셔너리 목록을 읽는다.
2. 각 딕셔너리를 `Quiz.from_dict()`에 넣는다.
3. 검증을 통과한 데이터만 `Quiz` 객체로 복원한다.

이 방식의 장점:

- 저장 형식은 JSON처럼 단순하다.
- 프로그램 내부에서는 객체처럼 편하게 다룰 수 있다.
- 저장 데이터가 손상되었을 때 검증 단계에서 문제를 빨리 발견할 수 있다.
- 데이터 표현과 게임 로직의 역할이 깔끔하게 나뉜다.

즉 `to_dict()`와 `from_dict()`는 "객체 세계"와 "JSON 세계"를 연결하는 다리 역할을 한다.

## 15. 입력 검증을 따로 메서드로 분리한 이유

`QuizGame` 안에는 `prompt_text()`와 `prompt_number()`가 있다.

이 메서드들을 만든 이유는 입력 규칙을 한곳에 모으기 위해서다.

예를 들어 숫자 입력에서는 아래를 반복해서 확인해야 한다.

- 공백 제거
- 빈 입력 검사
- 숫자인지 검사
- 범위 안인지 검사
- 취소 입력(`esc`)인지 검사

이 로직을 메뉴, 정답 입력, 삭제 번호 입력마다 매번 직접 쓰면 코드가 길어지고 중복이 많아진다.  
그래서 "검증 규칙"을 메서드로 빼 두면 나중에 수정도 훨씬 쉽다.

결과:

- 입력 처리 기준이 모든 기능에서 같아진다.
- 잘못된 입력을 받아도 프로그램이 쉽게 깨지지 않는다.
- 코드 리뷰 때도 "입력 검증은 여기서 한 번에 처리합니다"라고 설명할 수 있다.

## 16. 예외 처리를 왜 넣었는가

이 프로젝트는 정상 입력만 가정하지 않는다.  
실제로는 파일이 없을 수도 있고, 파일이 손상될 수도 있고, 사용자가 `Ctrl+C`를 누를 수도 있다.

그래서 다음 상황을 고려했다.

- `state.json`이 없으면 기본 퀴즈 사용
- `state.json`이 손상되면 기본 퀴즈로 복구
- 숫자가 아닌 입력이 들어오면 재입력 요청
- `KeyboardInterrupt`, `EOFError`가 발생하면 저장 후 안전 종료
- 저장 중 `OSError`가 발생하면 오류 메시지 출력

이렇게 한 이유는 "실패하지 않는 프로그램"보다 "문제가 생겨도 비정상 종료하지 않는 프로그램"을 만드는 것이 더 중요하기 때문이다.

결과:

- 콘솔 프로그램이 훨씬 안정적으로 동작한다.
- 초보자용 프로젝트이지만 실제 프로그램다운 방어 코드가 생긴다.

### 16-1. `try-except`문은 언제 사용하는가

`try-except`는 "오류가 날 가능성이 있는 코드"를 감싸고, 실제로 예외가 발생했을 때 프로그램이 바로 죽지 않도록 처리하는 문법이다.

언제 사용하는가:

- 파일을 읽거나 저장할 때처럼 외부 환경에 따라 실패할 수 있을 때
- 사용자의 입력이나 저장 데이터처럼 내가 100% 통제할 수 없는 값을 다룰 때
- 오류가 나더라도 프로그램을 종료하지 않고 복구하거나 안내 메시지를 보여 주고 싶을 때

이 프로젝트에서는 특히 저장 파일 처리에서 `try-except`가 중요하다.  
예를 들어 `state.json`이 손상되어 있으면 `json.load()` 과정에서 예외가 발생할 수 있다.  
이때 예외 처리가 없으면 프로그램이 바로 종료되지만, `try-except`가 있으면 기본 퀴즈로 복구하거나 오류 메시지를 보여 준 뒤 계속 실행할 수 있다.

또 저장 과정에서도 `OSError` 같은 예외가 발생할 수 있다.  
파일 경로 문제나 쓰기 실패가 생겨도 예외를 잡아 두면 사용자는 "왜 저장이 안 되었는지"를 알 수 있고, 프로그램도 더 안전하게 동작한다.

중요한 점은 `try-except`를 아무 데나 남용하지 않는 것이다.  
오류 가능성이 거의 없는 단순 계산까지 전부 감싸기보다는, "실제로 실패 가능성이 있고 그 실패를 내가 처리할 수 있는 부분"에만 사용하는 것이 좋다.

즉 `try-except`는 오류를 숨기기 위한 문법이 아니라, 예상 가능한 실패 상황에 대비해 프로그램을 더 안정적으로 만드는 안전장치다.

## 17. 데이터 파일 설명

저장 파일 경로는 프로젝트 루트의 `state.json`이다.  
역할은 퀴즈 목록과 최고 점수를 저장하는 것이다.

예시 스키마:

```json
{
    "quizzes": [
        {
            "question": "파이썬의 창시자는 누구인가요?",
            "choices": ["귀도 반 로섬", "리누스 토르발스", "제임스 고슬링", "브렌던 아이크"],
            "answer": 1
        }
    ],
    "best_score": 100,
    "best_correct_count": 5,
    "best_total_count": 5
}
```

필드 설명:

- `quizzes`: 저장된 퀴즈 목록
- `question`: 문제 문장
- `choices`: 4개의 선택지
- `answer`: 정답 번호
- `best_score`: 최고 점수
- `best_correct_count`: 최고 점수 당시 맞힌 개수
- `best_total_count`: 최고 점수 당시 전체 문제 수

이 구조를 선택한 이유는 사람이 직접 열어 봐도 읽기 쉽고, Python의 `dict`, `list`, `int`, `str` 구조와 자연스럽게 대응되기 때문이다.

## 18. `json.dump(data, file, ensure_ascii=False, indent=4)`를 왜 이렇게 썼는가

저장 코드에는 아래 구문이 들어 있다.

```python
json.dump(data, file, ensure_ascii=False, indent=4)
```

이 한 줄은 "Python 딕셔너리 데이터를 JSON 파일로 저장한다"는 의미다.

### 18-1. `json.dump()`는 무엇인가

`json.dump()`는 Python 데이터를 파일에 바로 JSON 형식으로 써 주는 함수다.

예를 들어 `data`가 이런 딕셔너리라면:

```python
{
    "best_score": 100,
    "quizzes": [
        {
            "question": "파이썬의 창시자는 누구인가요?",
            "choices": ["귀도 반 로섬", "리누스 토르발스", "제임스 고슬링", "브렌던 아이크"],
            "answer": 1
        }
    ]
}
```

`json.dump()`는 이 데이터를 `state.json`에 JSON 문자열 형태로 저장해 준다.

즉 이 함수는 "파이썬 객체 -> 파일 속 JSON 데이터"로 바꾸는 역할을 한다.

### 18-2. `data`는 무엇인가

첫 번째 인자인 `data`는 저장할 실제 내용이다.

이 프로젝트에서는 `save_state()` 안에서 이렇게 만든다.

```python
data = {
    "quizzes": [quiz.to_dict() for quiz in self.quizzes],
    "best_score": self.best_score,
    "best_correct_count": self.best_correct_count,
    "best_total_count": self.best_total_count,
}
```

즉 `data`는 현재 게임 상태 전체를 담은 딕셔너리다.

왜 딕셔너리로 만들었는가:

- JSON 구조와 자연스럽게 연결된다.
- 키 이름으로 데이터 의미를 바로 알 수 있다.
- 퀴즈 목록과 점수 정보를 함께 저장하기 쉽다.

### 18-3. `file`은 무엇인가

두 번째 인자인 `file`은 JSON 내용을 써 넣을 파일 객체다.

보통 앞에서 이런 식으로 파일을 연다.

```python
with self.state_path.open("w", encoding="utf-8") as file:
    json.dump(data, file, ensure_ascii=False, indent=4)
```

여기서 `file`은 실제 `state.json` 파일과 연결된 통로라고 생각하면 된다.

왜 필요한가:

- `json.dump()`는 "무엇을 저장할지"뿐 아니라 "어디에 저장할지"도 알아야 한다.
- 그래서 저장 대상 파일 객체를 함께 넘겨야 한다.

### 18-4. `ensure_ascii=False`는 왜 넣었는가

이 옵션은 한글 같은 비ASCII 문자를 그대로 저장하게 해 준다.

만약 이 옵션이 없거나 `True`라면 한글이 이런 식으로 저장될 수 있다.

```json
{
    "question": "\ud30c\uc774\uc36c\uc758 \ucc3d\uc2dc\uc790\ub294 \ub204\uad6c\uc778\uac00\uc694?"
}
```

이렇게 되면 프로그램은 읽을 수 있어도 사람이 파일을 직접 볼 때 매우 불편하다.

반면 `ensure_ascii=False`를 사용하면:

```json
{
    "question": "파이썬의 창시자는 누구인가요?"
}
```

처럼 한글이 그대로 저장된다.

왜 이 프로젝트에서 중요한가:

- 퀴즈 문제와 선택지가 한글이다.
- `state.json`을 직접 열어 봤을 때 사람이 읽을 수 있어야 한다.
- 학습용 프로젝트이므로 저장 파일도 이해 가능한 형태가 좋다.

즉 `ensure_ascii=False`는 "한글이 깨져 보이지 않도록" 하기 위한 선택이다.

### 18-5. `indent=4`는 왜 넣었는가

이 옵션은 JSON을 예쁘게 들여쓰기해서 저장한다.

만약 `indent`를 주지 않으면 JSON이 한 줄로 길게 저장될 수 있다.

예:

```json
{"best_score":100,"quizzes":[{"question":"파이썬의 창시자는 누구인가요?","choices":["귀도 반 로섬","리누스 토르발스","제임스 고슬링","브렌던 아이크"],"answer":1}]}
```

이렇게 저장되면 컴퓨터는 읽을 수 있지만 사람은 구조를 파악하기 어렵다.

반면 `indent=4`를 주면:

```json
{
    "best_score": 100,
    "quizzes": [
        {
            "question": "파이썬의 창시자는 누구인가요?",
            "choices": [
                "귀도 반 로섬",
                "리누스 토르발스",
                "제임스 고슬링",
                "브렌던 아이크"
            ],
            "answer": 1
        }
    ]
}
```

처럼 줄바꿈과 들여쓰기가 들어간다.

왜 좋은가:

- 사람이 파일 구조를 쉽게 읽을 수 있다.
- 저장된 데이터가 올바른지 직접 확인하기 쉽다.
- Git diff를 볼 때도 변경 내용을 파악하기 쉽다.

이 프로젝트는 학습용이기 때문에 저장 속도보다 가독성이 더 중요하다.  
그래서 `indent=4`를 사용했다.

### 18-6. 이 한 줄을 이렇게 정리할 수 있다

```python
json.dump(data, file, ensure_ascii=False, indent=4)
```

의 의미를 한 문장으로 정리하면:

"현재 게임 상태를 JSON 형식으로 `state.json`에 저장하는데, 한글은 그대로 보이게 하고, 사람이 읽기 쉽도록 들여쓰기까지 적용한다."

즉 이 코드는 단순 저장이 아니라, "사람도 읽을 수 있는 JSON 파일"을 만드는 설정까지 포함한 코드다.

## 19. 실행 결과로 얻은 것

이 프로젝트를 통해 실제로 확인할 수 있었던 점은 다음과 같다.

- 클래스는 데이터를 묶는 것뿐 아니라, 관련 동작까지 함께 묶을 때 의미가 커진다.
- JSON 저장은 단순 파일 저장이 아니라, "객체를 저장 가능한 형태로 변환하는 과정"까지 포함한다.
- 입력 검증을 제대로 하지 않으면 콘솔 프로그램은 생각보다 쉽게 깨진다.
- 예외 처리를 넣으면 사용자 경험이 훨씬 좋아진다.
- 코드를 너무 잘게 분리하면 오히려 흐름 파악이 어려워질 수 있다.
- 반대로 적당한 분리와 명확한 메서드 이름은 가독성을 크게 높인다.

즉 이 과제를 통해 "문법을 아는 것"과 "작동하는 프로그램을 끝까지 완성하는 것"은 다르다는 점을 직접 경험했다.
